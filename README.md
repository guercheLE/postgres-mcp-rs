# postgresql-mcp

PostgreSQL 18.4 catalog MCP (Model Context Protocol) server, generated by [mcpify](https://github.com/guercheLE/mcpify).

Exposes four tools — `search`, `get`, `call`, and `execute_sql`. Catalog discovery is backed by an embedded semantic database, while live calls use PostgreSQL's native wire protocol rather than HTTP.

`execute_sql` is the PostgreSQL counterpart to an `sp_executesql`-style escape hatch: it prepares exactly one statement and sends every supplied value separately through the extended query protocol. The tool permits reads and writes; the connected PostgreSQL role's grants are the authorization boundary.

## Install

```bash
cargo build --release
```

This builds three binaries into `target/release/`: `postgresql-mcp` (the CLI/server below), `postgresql-mcp-populate-embeddings`, and `postgresql-mcp-healthcheck`. Run `cargo install --path .` instead if you want `postgresql-mcp` on your `PATH` so the commands below work without a `target/release/` prefix.

Prebuilt binaries for macOS, Linux, and Windows are attached to each [GitHub Release](https://github.com/guercheLE/postgresql-mcp.git/releases), along with a shell/PowerShell installer script.

Or install the published crate directly:

```bash
cargo install postgresql-mcp
```

Registry publishing is enabled. A `v*.*.*` tag runs `cargo package --locked`, enforces the crates.io package size below 10,000,000 bytes, and then runs `cargo publish --locked` with `CARGO_REGISTRY_TOKEN`.

## Setup

```bash
cargo run -- setup
```

Interactively collects a PostgreSQL connection URL and username/password, then lets you persist them as a `.env` file, a local (`./postgresql-mcp.config.yml`) or global (`~/.postgresql-mcp/config.yml`) YAML config file, or a ready-to-run environment invocation.

The wizard rejects passwords embedded in the connection URL so they cannot leak into YAML configuration; enter the password only at its dedicated secret prompt. A wizard-created `.env` is permissioned `0600` on Unix.

Supported PostgreSQL authentication input: `password` (username/password). PostgreSQL negotiates SCRAM-SHA-256 or MD5 according to the database server's role and `pg_hba.conf` configuration.

## Configuration

| Env var | Purpose |
|---|---|
| `POSTGRES_MCP_DATABASE_URL` | PostgreSQL connection URL, for example `postgresql://db.example.com/app?sslmode=require`. |
| `POSTGRES_MCP_URL` | Backward-compatible alias for `POSTGRES_MCP_DATABASE_URL`. |
| `POSTGRES_MCP_USERNAME` / `POSTGRES_MCP_PASSWORD` | Overrides any stored PostgreSQL password credential — set both to authenticate without running `setup` first (checked before the OS keychain/encrypted-file fallback). |
| `POSTGRES_MCP_API_VERSION` | Embedded PostgreSQL catalog version used by `search`, `get`, and `call`. |
| `POSTGRES_MCP_LOG_LEVEL` | Log verbosity (`trace`/`debug`/`info`/`warn`/`error`). |

See `.env.example` for the full list of supported variables.

The URL is parsed by the native PostgreSQL driver. Username/password from the credential cascade override credentials embedded in the URL, so secrets can remain in the keychain or environment.

## Usage

### Terminal Client (default)

```bash
postgresql-mcp search "inspect active database connections"
postgresql-mcp get <operationId>
postgresql-mcp call <operationId> --args '{"key":"value"}'
postgresql-mcp execute-sql \
  'SELECT * FROM app.orders WHERE customer_id = $1::text::uuid' \
  --parameters '["7d92c2d4-d01a-4d80-8948-70b876582a7f"]'
```

`call` accepts one JSON object through `--args` (or `-a`), not arbitrary per-operation CLI flags. Use `get <operationId>` to see the accepted field names and which ones are required.

`execute-sql` accepts exactly one statement. Parameters use PostgreSQL's `$1`, `$2`, ... placeholders and have TEXT wire type; use explicit casts such as `$1::text::integer`. Do not concatenate caller-controlled values into the SQL text. SELECT-like statements return JSON rows and respect `--max-rows`; DML/DDL returns `affected_rows`.

Other subcommands: `postgresql-mcp test-connection` (connect and report database/user/server version), `postgresql-mcp config` (print the resolved configuration, secrets redacted), `postgresql-mcp version` (print the installed version), and `postgresql-mcp versions` (list the embedded catalog versions).

### Harness Server

```bash
postgresql-mcp start                              # stdio transport (default)
postgresql-mcp http --host 127.0.0.1 --port 3000  # HTTP transport
```

### Connect an MCP client

**stdio:** after running `postgresql-mcp setup`, configure an MCP host to spawn the server. Include the connection settings printed by the wizard:

```json
{
  "mcpServers": {
    "postgresql-mcp": {
      "command": "postgresql-mcp",
      "args": ["start"],
      "env": {
        "POSTGRES_MCP_DATABASE_URL": "postgresql://localhost:5432/your_database?sslmode=disable",
        "POSTGRES_MCP_AUTH_METHOD": "password",
        "POSTGRES_MCP_TRANSPORT": "stdio"
      }
    }
  }
}
```

Use the absolute executable path if `postgresql-mcp` is not on the MCP host's `PATH`. The stdio server reads the connection settings from this `env` block and uses the credentials saved by `setup`.

**HTTP:** every request must carry its own PostgreSQL username/password in an HTTP Basic `Authorization` envelope. Basic is only how the MCP HTTP transport carries the credentials; PostgreSQL still performs native password authentication, and the HTTP transport intentionally does not fall back to credentials stored on the server:

```json
{
  "mcpServers": {
    "postgresql-mcp": {
      "url": "http://127.0.0.1:3000/mcp",
      "headers": {
        "Authorization": "Basic <base64(username:password)>"
      }
    }
  }
}
```

Keep the listener on localhost unless you have added appropriate network access controls and TLS in front of it. Basic credentials are only encoded, not encrypted; use HTTPS when the MCP HTTP listener is exposed beyond localhost.

## Docker

```bash
# Stdio: the MCP client launches this one-off process and owns its stdin/stdout pipes
docker compose run --rm -T postgresql-mcp

# HTTP: a long-running network endpoint published on http://localhost:3000
docker compose up postgresql-mcp-http
```

Run these commands from the repository root. Docker Compose automatically discovers `docker-compose.yml`; `postgresql-mcp` and `postgresql-mcp-http` are service names inside that file, not filenames. Writing `docker compose -f docker-compose.yml ...` is equivalent, but `-f` is only needed when the file has another name or location, or when combining multiple Compose files.

Both services read configuration from a local `.env` file (copy `.env.example`) and persist credentials and configuration under `~/.postgresql-mcp` on the host. For stdio, `-T` disables pseudo-TTY allocation so MCP JSON-RPC stays on raw stdin/stdout, and `--rm` removes the one-off container when the client exits.

Stdio is a process transport, not a listening service: the MCP client must start the server and communicate through that exact child process's stdin/stdout. This is useful when an MCP client is configured to launch `docker compose run --rm -T postgresql-mcp`, in local scripts or CI that directly exchange MCP messages with the process, or in a custom image where your application launches the generated server's `start` subcommand as a child process. Merely putting the application and server in the same image—or starting the stdio container separately with `docker compose up`—does not connect their streams. One stdio server process normally serves one client. Use HTTP when independently started applications, multiple clients, another container, or a remote machine need to connect over the network.

## Observability & Resilience

### Logging

Structured logs go to **stderr** (never stdout, which is reserved for MCP JSON-RPC frames on stdio transport): JSON by default, pretty-printed automatically when stderr is an interactive TTY (auto-detected — there's no separate flag for this). Level is controlled by `POSTGRES_MCP_LOG_LEVEL` (default `info`), passed straight through to `tracing_subscriber::EnvFilter`, so directive syntax works too, e.g.:

```bash
POSTGRES_MCP_LOG_LEVEL="postgres_mcp=debug,warn" postgresql-mcp start
```

Secret redaction exists as a helper (`core::sanitizer::sanitize`, covering PostgreSQL passwords, authorization envelopes, secrets, credentials, and passwords embedded in connection URLs). Its caller is `postgresql-mcp config`, which prints the resolved config with those values redacted. Request/response payloads are not logged.

### OpenTelemetry tracing

An OTLP/HTTP trace exporter (`core/otel.rs`) is built unconditionally at startup; if it fails to build, tracing export is silently skipped — there's no dedicated on/off switch in this app. It's *tracing only* (no OTel metrics exporter is wired up — see "Metrics" below). Point it at a collector with the OTLP SDK's own standard env vars (not `POSTGRES_MCP_`-prefixed), which `opentelemetry-otlp` reads directly:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4318 postgresql-mcp start
```

Defaults to `http://localhost:4318` if unset. `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT`, `OTEL_EXPORTER_OTLP_HEADERS`, `_PROTOCOL`, `_TIMEOUT`, and `_COMPRESSION` are also honored (standard OTLP conventions).

### Metrics

Separate from OTel: `GET /metrics` (HTTP transport only — not available over stdio) serves a minimal hand-rolled Prometheus-text counter store (`http/metrics.rs`). Today it only tracks one counter, `http_requests_total`:

```bash
curl http://127.0.0.1:3000/metrics
# http_requests_total 4
```

### Native PostgreSQL execution and SQL injection

`call` translates generated catalog relation/function metadata into identifier-quoted SQL. Identifiers and cast types come only from the embedded store; operation arguments are bound separately.

`execute_sql` intentionally accepts read/write SQL, because PostgreSQL roles and grants—not an application-side keyword list—decide what the caller may change. Its injection boundary is strict:

- exactly one prepared statement per call; stacked statements are rejected by PostgreSQL;
- values travel separately as TEXT parameters through the extended query protocol;
- non-text values require explicit casts such as `$1::text::bigint`;
- statement execution uses `POSTGRES_MCP_TIMEOUT_MS` as PostgreSQL `statement_timeout`;
- row-producing statements default to 100 rows and are capped at 10,000.

The server does not automatically retry statements: retrying an ambiguous write could duplicate a committed side effect.

### Health checks

`GET /healthz` (HTTP transport only) reports the status of a `ComponentRegistry`, refreshed every 30 seconds with a 5-second per-check timeout by a `HealthCheckManager` — both intervals are hardcoded, not configurable. Today exactly one check is registered, `store` (can the active `mcp_store*.db` file be opened), marked critical:

```bash
curl http://127.0.0.1:3000/healthz
# {"status":"Healthy","components":1}   # 503 + "Unhealthy" if the critical check is failing
```

Two related but distinct checks exist:

- `postgresql-mcp-healthcheck` — the standalone binary wired into the Dockerfile's `HEALTHCHECK`; it only checks that the active store file exists and is readable on disk, and does not talk to a running server or `/healthz`.
- `postgresql-mcp test-connection` — an on-demand native PostgreSQL connection/authentication check; unrelated to the periodic `/healthz` checks above.

### Credential storage

`postgresql-mcp setup` writes credentials straight to the OS-native secret store via the `keyring` crate (macOS Keychain / Windows Credential Manager / Linux Secret Service), under service `postgresql-mcp`, account `active-credentials`. If no OS keychain backend is available (e.g. no D-Bus secret-service daemon in a minimal container), it falls back automatically to an AES-256-GCM-encrypted file at `~/.postgresql-mcp/credentials.enc` (`0600`, parent dir `0700` on Unix); the key is derived from `$HOME` plus the service name, so that file isn't portable to another machine.

The `POSTGRES_MCP_USERNAME`/`POSTGRES_MCP_PASSWORD` env vars documented in `.env.example` are read directly by `AuthManager::credentials()` and take priority over the stored keychain/file credentials — useful for supplying credentials purely via environment (e.g. in a container) without ever running `setup`.

Credentials always go through the keychain/encrypted-file path. If the operator explicitly chooses `.env`, that file also contains `POSTGRES_MCP_USERNAME` and `POSTGRES_MCP_PASSWORD` so it is immediately runnable; keep it untracked and permission-restricted. YAML config files contain only non-secret connection settings.

## Coverage

```bash
bash scripts/coverage.sh   # generates HTML and fails below 85% production-line coverage
```

The 85% gate counts executable production lines under `src/` and removes inline `#[cfg(test)]` module bodies from the LCOV denominator, so adding test code cannot inflate the result. The unfiltered annotated HTML remains useful for line-by-line analysis at `target/coverage/html/index.html`; the gate's machine-readable input is `target/coverage/production-lcov.info`. The command requires Python 3, `cargo-llvm-cov`, and the `llvm-tools-preview` Rust component.

## Profiling

```bash
bash scripts/profile.sh        # clean CPU profiling via samply
bash scripts/profile-heap.sh   # steady-state heap profiling via dhat-rs
```

CPU and heap profiling use separate builds: `scripts/profile.sh` deliberately profiles normal release binaries so DHAT allocation tracking cannot distort CPU samples, while `scripts/profile-heap.sh` starts DHAT collection only after its warmup search. CPU profiling records `profile/cold-start.json.gz` from a one-shot CLI search, then attaches to an already-initialized search harness and records `profile/warm-search.json.gz`; this keeps model initialization from being mistaken for steady-state request cost. Heap profiling defaults to 1 warmup and 5 measured searches, configurable with `PROFILE_HEAP_WARMUPS`, `PROFILE_HEAP_ITERATIONS`, and `PROFILE_QUERY`. Both scripts supply harmless connection defaults when a generated checkout has not been configured because catalog search never connects to PostgreSQL. `profile/bottleneck-report.md` ranks coverage gaps and shows separate cold and warm CPU summaries. Requires [samply](https://github.com/mstange/samply) (`cargo install samply`).
